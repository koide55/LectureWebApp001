# Web アプリケーション脆弱性演習
## 発展演習課題集

この資料は、講義内で扱った基本演習に加えて取り組むための発展演習課題集である。  
既存アプリの挙動を観察するだけでなく、コードを読んで変更し、安全な実装や脆弱な実装の違いをより深く理解することを目的とする。

## 使い方

- 基本演習を終えた後に取り組むこと
- すべてを実施する必要はない
- 難易度に応じて、初級・中級・上級から選択すること
- 課題によってはコード変更、動作確認、簡単な説明文の作成を求める

## 提出の基本方針

- 変更したファイル名を示すこと
- 何を変更したかを簡潔に説明すること
- どの脆弱性または対策に関係するかを書くこと
- 動作確認結果を短くまとめること

## 推奨提出形式

1. 課題番号
2. 変更したファイル
3. 実装内容
4. 動作確認結果
5. 気づいたこと

## Moodle提出指示一覧

| 課題 | 回答形式 | 必ず含める項目 |
|---|---|---|
| 初級 1-4 | 作文 | 課題番号、対象ファイル、答えまたは比較結果 |
| 中級 1-4 | ファイル提出 + 作文 | 課題番号、変更ファイル、実装内容、動作確認結果 |
| 上級 1-4 | ファイル提出 + 作文 | 課題番号、変更ファイル、設計意図、動作確認結果、対策説明 |

作文の書き方:

- 初級: 120-220字を目安に、番号付きで答える
- 中級: 200-400字を目安に、実装内容と確認結果を分けて書く
- 上級: 250-500字を目安に、問題・実装・対策を分けて書く

## 初級

初級では、既存コードの読解と小さな修正を中心に行う。

### 初級 1 `role_required` の読解と説明

目的:

- 認可チェックの基本を理解する

課題:

- `role_required` の処理を読み、未ログイン時と権限不足時で何が起こるか説明する
- `/admin` がなぜ保護されているかを説明する

対象候補:

- `app/auth/decorators.py`
- `app/routes.py`

提出物:

- 処理の説明文
- 関連するコード位置

### 初級 2 SQLインジェクション対策の確認

目的:

- 安全版と脆弱版の違いをコードレベルで理解する

課題:

- `/users` に関する safe / vulnerable の実装差分を確認する
- なぜプレースホルダが有効か説明する

対象候補:

- `app/services/user_service.py`

提出物:

- safe / vulnerable の差分説明
- 危険な点と安全な点の整理

### 初級 3 XSS 対策の確認

目的:

- 出力時のエスケープの意味を理解する

課題:

- `/reflect` と `/board` のテンプレートを読み、どこで自動エスケープが効くか確認する
- `|safe` がなぜ危険か説明する

対象候補:

- `app/templates/reflect.html`
- `app/templates/board.html`

提出物:

- コード上の比較
- 反射型 XSS と蓄積型 XSS の違いの説明

### 初級 4 コマンドインジェクション対策の確認

目的:

- `shell=True` と引数配列の違いを理解する

課題:

- `safe_ping()` と `unsafe_ping()` を比較する
- どの行が危険か、どの行が対策になっているかを書く

対象候補:

- `app/services/command_service.py`

提出物:

- 比較表
- 危険な実装の説明

## 中級

中級では、小さな機能追加や既存機能の安全化を行う。

### 中級 1 新しい保護ページの追加

目的:

- 認証と認可を使ったページ保護を自分で実装する

課題:

- 例として `/teacher` や `/staff` のような新しいページを追加する
- ログイン済みで、かつ特定ロールだけがアクセスできるようにする

条件:

- サーバ側で保護すること
- ルートとテンプレートを追加すること

提出物:

- 追加したページの説明
- 使ったデコレータの説明
- 動作確認結果

### 中級 2 プロフィール更新機能の強化

目的:

- 状態変更処理と CSRF 保護の意味を理解する

課題:

- `/profile` に新しい項目を1つ追加する
- CSRF 保護 ON のときに正しく動くようにする
- 表示側でも安全に扱う

例:

- 自己紹介の別項目
- 所属
- ニックネーム

提出物:

- 追加した項目
- 更新処理の説明
- CSRF と出力安全性の観点からの説明

### 中級 3 管理者専用機能の追加

目的:

- 認可不備を防ぐ設計を考える

課題:

- `/admin` 配下に新しい機能を1つ追加する
- 例として、投稿一覧表示、投稿削除、ユーザ一覧表示などを実装する
- 一般ユーザがその機能を使えないことを確認する

条件:

- 管理者ロールが必要であること
- 一般ユーザでアクセスしたときの挙動を確認すること

提出物:

- 追加機能の説明
- 認可チェックの説明
- 動作確認結果

### 中級 4 新しい検索機能の追加

目的:

- SQL を安全に扱う実装を自分で作る

課題:

- 新しい検索ページを追加する
- 例として、投稿検索やプロフィール検索などを実装する
- SQL を安全に書くこと

発展:

- safe / vulnerable の両方を作って比較してもよい

提出物:

- 検索対象
- SQL の書き方
- 安全性についての説明

## 上級

上級では、新しい脆弱性題材の追加や、複数の観点を組み合わせた設計を行う。

### 上級 1 IDOR の実装と対策

目的:

- 認可不備をより具体的に理解する

課題:

- 他人のデータにアクセスできてしまうページを追加する
- その後、安全版へ修正する

例:

- `/posts/<id>`
- `/profile/<id>`

観点:

- URL の値だけで他人の情報にアクセスできないようにする
- 所有者または管理者だけが操作できるようにする

提出物:

- 脆弱版の説明
- 対策版の説明
- 何を認可すべきかの整理

### 上級 2 発展的な XSS 演習の追加

目的:

- 出力位置の違いによる危険性の違いを理解する

課題:

- 既存の XSS 演習に加えて、別の出力位置を持つ表示機能を追加する
- どのような出力文脈で危険になるか考える

例:

- 属性値への出力
- JavaScript 文字列への埋め込み

提出物:

- 追加した機能
- なぜ危険か
- どう防ぐべきか

### 上級 3 複合脆弱性シナリオの作成

目的:

- 脆弱性を単独ではなく組み合わせて考える

課題:

- 2種類以上の脆弱性が関係するシナリオを設計する
- 例として、XSS と CSRF、認可不備と IDOR などを組み合わせる
- 想定される攻撃と対策を説明する

提出物:

- シナリオ説明
- 関係する機能
- 攻撃の流れ
- 対策の流れ

### 上級 4 セキュリティレビュー課題

目的:

- 自分で脆弱性を探す力を身につける

課題:

- この教材アプリ全体を対象に、脆弱性または改善点を 3 つ以上挙げる
- 既存の講義で扱った内容でもよいが、コード位置を明示すること
- 修正方針も書くこと

提出物:

- 問題点の一覧
- 該当箇所
- 修正方針

## 発展演習の進め方

おすすめの進め方:

1. まず既存コードを読む
2. 変更前の動作を確認する
3. 小さく修正する
4. 再度動作を確認する
5. 変更内容と理由を説明する

## 評価の観点

- 脆弱性や対策の理解ができているか
- コード変更の意図が明確か
- 動作確認ができているか
- 説明が論理的か

## 注意

- 演習はローカル環境または授業用環境でのみ行うこと
- 意図的に脆弱なコードを作る場合も、公開環境では実行しないこと
- 他人の PC や公開サーバに対して試さないこと

## 参考

- 基本演習は講義スライドと通常教材を参照すること
- 発展演習では、safe / vulnerable の差分を自分で説明できることを目指すこと
