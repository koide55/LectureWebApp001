# Web アプリケーション脆弱性演習
## 発展演習ハンドアウト

この資料は、通常の講義内演習に加えて取り組むための発展演習用ハンドアウトである。  
既存アプリの挙動を観察するだけでなく、コードを読み、変更し、動作を確認しながら理解を深めることを目的とする。

## 目的

- 脆弱性と対策をコードレベルで理解する
- 安全な実装と危険な実装の違いを説明できるようにする
- 小さな機能追加や修正を通じて、実装上の判断を体験する

## 取り組み方

1. まず基本演習を終える
2. 変更前の挙動を確認する
3. 対象コードを読む
4. 小さく変更する
5. 動作確認を行う
6. 変更内容と理由をまとめる

## 難易度

### 初級

- 既存コードを読む
- safe / vulnerable の違いを説明する
- 小さな修正を行う

### 中級

- 新しい小機能を追加する
- 認証、認可、CSRF、出力安全性を意識して実装する

### 上級

- 新しい脆弱性シナリオを追加する
- 複数の観点を組み合わせて設計する
- セキュリティレビューの形で改善点を提案する

## 提出方法

提出時は、次の項目を含めること。

1. 課題番号
2. 変更したファイル
3. 実装内容の要約
4. 動作確認結果
5. 学んだこと

## 評価の観点

- 脆弱性または対策の理解が正しいか
- コード変更の意図が明確か
- 動作確認ができているか
- 説明が簡潔で論理的か

## Moodle提出指示一覧

| 課題 | 回答形式 | 必ず含める項目 |
|---|---|---|
| B1-B4 | 作文 | 課題番号、対象ファイル、答えまたは比較結果 |
| I1-I4 | ファイル提出 + 作文 | 課題番号、変更ファイル、実装内容、動作確認結果 |
| A1-A4 | ファイル提出 + 作文 | 課題番号、変更ファイル、設計意図、動作確認結果、対策説明 |

作文の書き方:

- 初級: 120-220字を目安に、番号付きで答える
- 中級: 200-400字を目安に、実装内容と確認結果を分けて書く
- 上級: 250-500字を目安に、問題・実装・対策を分けて書く

## 初級課題

### B1 `role_required` の読解と説明

目的:

- 認可チェックの基本を理解する

作業:

- `role_required` の動作を説明する
- 未ログイン時と権限不足時の挙動を整理する
- `/admin` がどのように保護されているか説明する

対象ファイル:

- `app/auth/decorators.py`
- `app/routes.py`

### B2 SQLインジェクション対策の比較

目的:

- safe / vulnerable の差分をコードレベルで理解する

作業:

- `/users` に関する実装を読む
- 脆弱版の危険な点を説明する
- 安全版で何が改善されているか説明する

対象ファイル:

- `app/services/user_service.py`

### B3 XSS 対策の比較

目的:

- 出力時のエスケープの重要性を理解する

作業:

- `/reflect` と `/board` のテンプレートを読む
- `|safe` の有無で何が変わるか説明する
- 反射型 XSS と蓄積型 XSS の違いを整理する

対象ファイル:

- `app/templates/reflect.html`
- `app/templates/board.html`

### B4 コマンドインジェクション対策の比較

目的:

- `shell=True` が危険な理由を理解する

作業:

- `safe_ping()` と `unsafe_ping()` を比較する
- どの行が危険で、どの行が対策か説明する

対象ファイル:

- `app/services/command_service.py`

## 中級課題

### I1 新しい保護ページの追加

目的:

- 認証と認可を使ったページ保護を実装する

作業:

- `/teacher` または `/staff` のような新しいページを追加する
- 特定ロールだけがアクセスできるようにする
- サーバ側で保護する

### I2 プロフィール更新機能の拡張

目的:

- 状態変更処理と CSRF 保護の意味を理解する

作業:

- `/profile` に新しい項目を追加する
- CSRF 保護 ON でも動くようにする
- 表示時の安全性にも注意する

### I3 管理者専用機能の追加

目的:

- 認可不備を防ぐ設計を考える

作業:

- `/admin` 配下に新機能を追加する
- 一般ユーザでは使えないことを確認する

例:

- 投稿一覧
- 投稿削除
- ユーザ一覧

### I4 新しい検索機能の追加

目的:

- SQL を安全に扱う機能を自分で実装する

作業:

- 新しい検索ページを追加する
- SQL を安全に書く
- 必要なら safe / vulnerable を両方用意して比較する

## 上級課題

### A1 IDOR の実装と対策

目的:

- 認可不備をより具体的に理解する

作業:

- 他人のデータにアクセスできてしまうページを追加する
- その後、安全版に修正する

例:

- `/posts/<id>`
- `/profile/<id>`

### A2 発展的な XSS 演習の追加

目的:

- 出力文脈によって危険性が変わることを理解する

作業:

- 既存の XSS 演習以外に、別の出力位置を持つ表示機能を追加する
- なぜ危険か、どう防ぐかを説明する

### A3 複合脆弱性シナリオの設計

目的:

- 脆弱性を組み合わせて考える

作業:

- 2つ以上の脆弱性が関係するシナリオを設計する
- 攻撃の流れと対策の流れを説明する

### A4 セキュリティレビュー

目的:

- 自分で問題点を見つける力を身につける

作業:

- アプリ全体から脆弱性または改善点を 3 つ以上挙げる
- 該当箇所と修正方針を書く

## 注意事項

- 演習はローカル環境または授業用環境でのみ行うこと
- 意図的に脆弱なコードを作る場合も、公開環境では実行しないこと
- 公開サーバや他人の環境に対して試さないこと
